# 电商购物平台 - 项目开发总结报告

## 1. 项目简介

本项目为课堂贯穿项目「电商购物平台」的延续与完善，采用 **Vue 3 + Spring Boot + MySQL** 前后端分离架构，实现了用户注册登录、商品浏览与搜索、购物车、订单生成与模拟支付、订单查询与取消、商品评价以及管理员用户管理与评价管理等功能，满足作业对技术选型、功能模块数量与分层架构的要求。

---

## 2. 开发过程简述

### 2.1 需求与设计

- 依据作业要求与常见电商业务流程，确定用户管理、商品管理、购物车、订单管理、支付（模拟）、评价等模块，并补充管理员侧的用户管理与评价管理。
- 采用分层架构：表示层（Controller + 前端）、业务逻辑层（Service）、数据访问层（Mapper）、实体层（Entity/DTO）、通用层（Result、异常处理、安全配置）。
- 数据库使用 MySQL，设计 user、category、product、cart_item、order_main、order_item、review 等表，并编写建表与种子数据脚本。

### 2.2 技术实现要点

- **认证与权限**：使用 JWT 进行无状态认证，Spring Security 放行登录/注册及公开接口，其余接口需携带 Token；管理员接口通过角色限制。
- **事务与一致性**：下单、支付、取消等涉及多表写操作的方法使用 `@Transactional`，保证数据一致。
- **异常与校验**：使用 `GlobalExceptionHandler` 统一处理业务异常与参数校验异常，返回统一 `Result` 结构；入参使用 Jakarta Validation 进行校验。
- **部署**：使用 Docker Compose 一键启动前端、后端与 MySQL，便于演示与验收。

### 2.3 文档与测试

- 在开发与验收阶段补充了需求分析说明书、系统设计说明书（含 UML 类图、用例图、时序图）、数据库设计说明（含 ER 图）、用户操作手册、测试报告与项目开发总结报告。
- 使用 JUnit 5 与 Mockito 编写 Service 层单元测试，使用 MockMvc 编写认证相关接口的集成测试；系统测试通过手工操作完成。

## 3. 开发过程中遇到的问题及解决方案

### 3.1 前后端跨域与 Context Path

**问题**：前端部署在 Nginx，请求发往 `/api`，后端 context-path 为 `/api`，若 Nginx 代理未正确转发，会出现 404 或跨域问题。

**解决**：在 Nginx 配置中将 `/api` 反向代理到后端服务，并统一前端 axios baseURL 为相对路径或 `/api`；后端开启 CORS，允许前端域名与常用方法、头信息。

### 3.2 订单号唯一性与并发

**问题**：订单号需唯一且可读，若仅用时间戳在高并发下可能重复。

**解决**：采用「时间戳 + 自增序号」生成订单号（如 `O + yyyyMMddHHmmss + 4 位序号`），在 Service 内使用 `AtomicInteger` 做自增，在单机下可避免重复；若后续多实例部署，可改为分布式序号或 UUID 后缀。

### 3.3 购物车勾选与结算一致性

**问题**：结算时若只根据「当前勾选」查询，可能在提交瞬间用户取消勾选或库存变化，导致数据不一致。

**解决**：在创建订单时再次从数据库查询当前用户的勾选项与商品库存，校验通过后再写订单、扣库存、删购物车项，并在同一事务中完成，减少不一致窗口。

### 3.4 测试环境与生产数据库差异

**问题**：单元测试若依赖真实 MySQL，需要维护测试库且速度较慢；集成测试若用 MySQL，CI 或本机环境可能未安装 MySQL。

**解决**：单元测试使用 Mockito 模拟 Mapper，不依赖数据库；集成测试使用 H2 内存库（MODE=MySQL）并执行 H2 兼容的建表脚本（schema-h2.sql），保证无外网依赖、执行快速。若 H2 与 MyBatis 中部分 SQL 存在方言差异，可在文档中说明并在本地用 MySQL 验证关键流程。

### 3.5 UML 与文档维护

**问题**：手绘 UML 图在代码变更后容易过时，且不便于版本管理。

**解决**：使用 PlantUML 编写类图、用例图、时序图与 ER 图（.puml 文件），放在 `docs/uml/` 下，与文档一起纳入版本控制；文档中注明「图文件位置」与生成图片的方式（如 PlantUML 插件或命令行），便于评审与后续更新。

## 4. 心得体会

1. **分层与职责**：严格区分 Controller、Service、Mapper 的职责，便于单元测试（Mock Mapper）和后续扩展（如更换持久层），也符合单一职责与依赖倒置的指导思想。
2. **统一响应与异常**：使用统一 `Result` 与全局异常处理，前端处理更简单，接口风格一致，便于联调与排查问题。
3. **文档与代码同步**：需求分析、系统设计、数据库设计、用户手册等文档与代码同步更新，能减少理解偏差，也满足作业对文档完整性的要求。
4. **测试的价值**：单元测试在重构与加功能时能快速发现回归；集成测试能验证请求链路与安全配置；手工系统测试能发现交互与业务细节问题，三者结合能提升交付质量。
5. **工程化与交付**：Docker Compose 与 README 中的启动说明使项目易于在他人环境中运行，有利于答辩与验收。

---

## 5. 不足与改进方向

- **测试覆盖**：当前集成测试仅覆盖认证接口，后续可增加订单、购物车、评价等接口的集成测试；可考虑引入覆盖率插件（如 JaCoCo）并设定覆盖率目标。
- **面向对象体现**：实体与 DTO 以数据为主，业务层继承与多态使用较少；若课程强调 UML 与设计原则，可在文档中补充对「开闭、依赖倒置」的体现说明，或对部分 Service 增加接口抽象以便扩展与测试。
- **性能与安全**：未做专门的压力测试；生产环境需注意 JWT 密钥与敏感配置的安全、SQL 注入防护（MyBatis 参数化已基本规避）、以及接口限流与防刷等。

---

## 6. 总结

本项目在技术选型、功能模块、分层架构、数据库设计、部署方式以及文档与测试等方面均按作业要求完成，并在开发过程中对跨域、订单号、事务一致性、测试环境与文档维护等问题进行了针对性解决。通过本次项目，进一步巩固了 Java 面向对象编程、Spring Boot 与 MyBatis 的使用、前后端分离与 JWT 认证、以及文档与测试在工程实践中的重要性。
